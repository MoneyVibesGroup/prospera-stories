## User Story

En tant qu'**administrateur plateforme Money Vibes**,
je veux **octroyer, mettre à jour ou révoquer les droits d'usage (entitlements) d'un module pour une organisation**,
afin d'**« allumer » les modules que le client peut utiliser (ex. Bilan avec le référentiel SYSCOHADA)**.

---

## Contexte

**Le geste central de la tour de contrôle** : octroyer un entitlement `ACTIVE` pour `(org × module × référentiel/version)` déclenche `entitlement.changed` (STORY-034) → le read-model de l'app cliente s'actualise → **le module apparaît** chez l'utilisateur (FE-014). Sous dogfooding (DG-1), c'est **ici** qu'on active le Bilan pour Money Vibes, **sans passer par le paiement**.

---

## Périmètre

**Inclus :**
- Depuis la fiche org (AP-02) : **liste des entitlements** de l'org (module, version, référentiel, statut ACTIVE/REVOKED).
- **Octroyer** : choisir un module + version + référentiel (depuis le catalogue AP-04) → grant `ACTIVE`.
- **Mettre à jour** (version/référentiel) et **révoquer** un entitlement.
- **Vue de réconciliation** (état effectif) + **validation contre le catalogue** (pas d'octroi d'un couple inexistant).
- Feedback + rafraîchissement ; note de cohérence éventuelle (l'app cliente suit via l'événement).

**Hors périmètre :**
- Définition du catalogue (AP-04).
- Séquence de provisioning multi-modules d'un vertical (AP-06, qui réutilise cette story).
- Facturation/plans (Module 2).

**Décisions & vigilance :**
- **Effet asynchrone** : l'UI confirme l'octroi côté catalogue ; la disponibilité chez le client suit l'événement (ne pas promettre un effet instantané).
- **Cohérence catalogue** : proposer uniquement des modules/versions/référentiels existants (source AP-04) pour éviter les grants invalides.
- **Traçabilité** : chaque action (grant/revoke) est un acte admin — afficher l'auteur/horodatage si l'API les fournit.
