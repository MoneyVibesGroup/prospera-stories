## User Story

En tant qu'**administrateur plateforme Money Vibes**,
je veux **activer un vertical (cabinet / IMF / distributeur / assurance) pour une organisation en une opération guidée**,
afin de **provisionner d'un coup l'ensemble des modules et le référentiel adaptés à ce type de client**.

---

## Contexte

Là où AP-05 octroie **un** entitlement, AP-06 orchestre une **séquence** : « activer le vertical cabinet pour l'org X » = octroyer le jeu de modules initiaux + le référentiel adéquat, après contrôle des pré-requis (KYC `APPROVED`). C'est la brique qui **prépare l'ouverture des verticaux** (P8) et rend le dogfooding reproductible. Elle réutilise l'octroi d'entitlement (AP-05).

> Le **déploiement** d'une instance isolée (ex. IMF sur son cloud) reste **hors v1** — ici on provisionne les **droits**, pas l'infrastructure (à cadrer JIT).

---

## Périmètre

**Inclus :**
- **Assistant de provisioning** : choisir un **vertical** (cabinet/imf/distributeur/assurance) pour une org → **jeu de modules initiaux** + **référentiel** proposés (config par vertical).
- **Contrôle des pré-requis** : KYC `APPROVED` requis ; sinon blocage explicite (renvoi vers AP-03).
- **Exécution** : séquence d'octrois d'entitlements (réutilise AP-05) ; résultat récapitulé ; **trace** de l'action.
- Config déclarative des « packs vertical » (modules + référentiel par défaut), extensible.

**Hors périmètre :**
- Déploiement/instanciation d'infrastructure (stack dédiée-vs-mutualisée, JIT).
- Facturation (Module 2).

**Décisions & vigilance :**
- **Idempotence** : provisionner deux fois ne doit pas dupliquer les entitlements ; gérer les octrois déjà présents.
- **Pré-requis** : le KYC `APPROVED` est un garde-fou métier (ne pas activer un client non validé).
- **Packs config-driven** : les jeux de modules par vertical vivent en config (alignés sur `vertical_backlog` du tracker programme).
