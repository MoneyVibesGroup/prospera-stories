## User Story

En tant que **super-admin de compte (`TENANT_ADMIN`)**,
je veux **gérer les utilisateurs de mon cabinet (lister, inviter, changer de rôle, suspendre, supprimer)**,
afin de **contrôler qui accède à l'espace du cabinet et avec quels droits**.

---

## Contexte

Écran d'administration interne au cabinet (FR-004). Réservé au rôle `TENANT_ADMIN` (un `TENANT_USER` reçoit 403 côté backend — l'UI masque l'entrée et gère le refus). Les endpoints sont servis par l'IdP (`auth-service`) et n'exposent que les utilisateurs **du tenant courant** (isolation garantie côté serveur par le `tenantId` du token — jamais fourni par le client).

**Garde-fou critique (FR-004) :** impossible de supprimer ou rétrograder le **dernier `TENANT_ADMIN` actif**. Le backend est la source de vérité (STORY-026 l'impose transactionnellement) ; l'UI doit **anticiper** ce cas (désactiver l'action + expliquer) et **gérer proprement** le rejet 409 si la course est perdue.

---

## Périmètre

**Inclus :**
- **Liste paginée** des utilisateurs du cabinet (nom, e-mail, rôle, statut `INVITED/ACTIVE/SUSPENDED`) ; pagination (défaut 20) ; états loading/error/empty (FE-003).
- **Inviter** : dialog (e-mail + rôle `TENANT_ADMIN`/`TENANT_USER`) → `POST /users` ; l'invité apparaît `INVITED` ; feedback succès.
- **Changer le rôle** : action par ligne → `PATCH /users/:id` ; confirmation.
- **Suspendre / réactiver** : `PATCH /users/:id` ; confirmation.
- **Supprimer** (soft delete) : `DELETE /users/:id` ; confirmation.
- **Garde-fous UI** : dernier `TENANT_ADMIN` actif → actions supprimer/rétrograder **désactivées** avec explication ; gestion du 409 backend (course perdue) via toast clair.
- **RBAC UI** : écran accessible aux seuls `TENANT_ADMIN` (entrée de menu masquée sinon) ; 403 backend géré (redirection/message).
- Invalidation TanStack Query après chaque mutation (liste à jour).

**Hors périmètre :**
- Acceptation de l'invitation par l'invité → **FE-007**.
- Édition du profil personnel de l'utilisateur courant.
- Revue/administration inter-tenant (persona plateforme) → hors app.

**Décisions & vigilance :**
- **Isolation** : ne jamais envoyer de `tenantId` depuis le client — le backend l'injecte depuis le token (NFR-002). L'UI n'affiche que ce que l'API renvoie.
- **Dernier admin** : calculer côté UI « nombre d'admins actifs » pour désactiver proactivement, MAIS toujours gérer le 409 (la source de vérité est le backend, la course existe).
- **Rôle courant** : lire le rôle depuis la session (FE-005) pour le RBAC d'affichage ; le refus réel reste 403 backend.
- **Soft delete** : un utilisateur supprimé disparaît de la liste active ; ne pas exposer de réactivation d'un supprimé (non réversible côté STORY-026).
- Réutiliser `Dialog`, `AlertDialog`, `Table`, `Badge` de shadcn.
