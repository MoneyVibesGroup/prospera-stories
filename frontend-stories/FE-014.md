## User Story

En tant qu'**utilisateur d'un cabinet (TENANT_ADMIN / TENANT_USER)**,
je veux **voir sur ma page d'accueil les modules qui me sont réellement ouverts et l'état de ceux qui ne le sont pas**,
afin de **savoir immédiatement ce que je peux utiliser (ex. Bilan) et ce qui me manque pour débloquer le reste**.

---

## Contexte

Cette story pose le **socle config-driven** de l'app cliente : l'app n'affiche que les **modules `ACTIVE`** pour l'organisation, en lisant l'**entitlement** (read-model alimenté par `entitlement.changed` du `platform-catalog-service`, octroyé depuis l'**admin-panel**). C'est le pendant client de la tour de contrôle : *l'admin allume un module → il apparaît ici*.

Le gate d'accès effectif combine trois conditions (déjà exposées côté backend) : `emailVerified` + KYC `APPROVED` + entitlement `ACTIVE` (l'abonnement viendra avec le Module 2). Le frontend **reflète** cet état et **oriente** — il ne décide jamais de l'autorisation (source de vérité = backend, 403 explicites).

> Prépare l'arrivée du module **Bilan** (FE-B00+) et de tout futur module/vertical : ajouter un module = le déclarer + le brancher, l'affichage suit l'entitlement.

---

## Périmètre

**Inclus :**
- **Registre de modules côté client** (config déclarative : clé, libellé, icône, route, entitlement requis) — extensible par vertical via `NEXT_PUBLIC_VERTICAL`.
- **Home des modules** : grille de cartes, chaque module `ACTIVE` = accessible (CTA d'entrée), sinon **verrouillé** avec la raison (`KYC_NOT_APPROVED`, `SUBSCRIPTION_REQUIRED`, entitlement absent) + CTA vers l'étape à traiter.
- **Lecture de l'état** : `GET /ec/tenant` (email/KYC/abonnement, déjà en FE-010) + **entitlements** de l'org (via `/ec/modules` ou `/catalog` — voir Notes) ; TanStack Query.
- **Navigation config-driven** : la nav latérale (FE-003) ne montre que les modules ouverts ; un module verrouillé n'apparaît pas (ou grisé selon la maquette).
- **Cohérence 403** : réutilise `accessErrorTarget`/`useAccessErrorRedirect` (FE-010) pour orienter quand un module gardé renvoie 403.

**Hors périmètre :**
- Le contenu des modules eux-mêmes (Bilan = FE-B00+).
- L'octroi/révocation d'entitlement → **admin-panel** (AP-05).
- Paiement/abonnement (FE-011/012/013).

## Critères d'acceptation

- [ ] Un registre de modules déclaratif existe ; l'ensemble affiché dérive de la config du vertical courant (aucune liste en dur dans les composants d'UI).
- [ ] La home liste les modules ; un module dont l'entitlement est `ACTIVE` (et pré-requis KYC OK) est **ouvert** (CTA d'entrée fonctionnel).
- [ ] Un module non ouvert est **verrouillé** avec la **raison** affichée (KYC non approuvé / abonnement requis / non attribué) + CTA contextuel (vers `/kyc`, onboarding, ou message « contactez votre administrateur »).
- [ ] La navigation ne propose que les modules ouverts ; l'accès direct à une route de module non ouvert redirige/oriente (pas d'erreur brute).
- [ ] L'état est rafraîchi à l'entrée de l'espace `(app)` et après les actions clés (approbation KYC, changement d'entitlement) ; cohérence éventuelle tolérée (staleTime + invalidation).
- [ ] i18n FR ; tests : mapping (état + entitlements) → modules ouverts/verrouillés + raisons ; cas 403 orienté.
