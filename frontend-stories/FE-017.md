## User Story

En tant qu'**équipe**,
je veux **un test automatisé qui prouve qu'un octroi fait dans la console admin allume réellement le module dans l'app cliente**,
afin de **ne jamais casser en silence la boucle qui justifie toute notre topologie**.

---

## Contexte

**C'est la story qui rend le principe 0 exécutable.** Le sprint 5 livre les deux moitiés d'une même
preuve : l'admin octroie (AP-05), le client s'allume (FE-014). Chacune, seule, est indémontrable — et
surtout, **rien ne garantit qu'elles restent connectées** après un refactor de l'une ou de l'autre.

C'est le **miroir frontend** du `npm run test:chain` backend (STORY-049), qui a fait passer la chaîne KYC
du statut « ça a marché une fois en démo » à « c'est prouvé rejouable ». La même logique s'applique ici :
la boucle entitlement est le cœur du dogfooding **DG-1** (l'entitlement est octroyé **manuellement** via
l'admin-panel tant que le paiement n'existe pas, sprint 16) et de la topologie **Option B** (une seule app
cliente config-driven qui n'affiche que les modules actifs).

Sans ce test, la première régression silencieuse se découvre chez le client zéro — c'est-à-dire nous.

---

## Portée

**Playwright, contre le stack docker réel** (`docker compose up`, 4 services + mongo/kafka), pas de mock.

Scénario nominal :

1. **Arrange** — une org cabinet KYC-approuvée, **sans** l'entitlement `bilan` (état de départ explicite,
   pas hérité d'un test précédent).
2. **Client** : se connecter en `TENANT_ADMIN` → le module n'apparaît **pas** dans la home des modules.
3. **Admin** : se connecter en `PLATFORM_ADMIN` → octroyer l'entitlement `bilan@<version>` à cette org.
4. **Propagation** : `entitlement.changed` (outbox → Kafka, clé `orgId`, état **absolu**).
5. **Client** : recharger → le module `bilan` **apparaît**.
6. **Révocation** : l'admin révoque (soft → `REVOKED`) → le module **disparaît** côté client.

⚠️ **Le piège à traiter, pas à ignorer : la propagation est ASYNCHRONE.** Le grant est synchrone côté
catalog, mais la diffusion passe par l'outbox relay (`relayIntervalMs` ≈ 2 s) puis Kafka. Un
`waitForTimeout` arbitraire rendra ce test **instable** — c'est la façon la plus sûre de le faire
désactiver dans trois sprints. Utiliser une attente **sur condition** (polling de l'UI / de
`GET /catalog/entitlements/:orgId` jusqu'à convergence, avec timeout généreux et message d'échec clair).

---

## Critères d'acceptation

1. Le test s'exécute contre le stack docker réel, **sans aucun mock**, et est **rejouable** (état de
   départ construit par le test, pas supposé).
2. Il échoue de façon **lisible** si la boucle casse (message qui dit quelle étape n'a pas convergé).
3. Il couvre l'octroi **et** la révocation (l'état absolu de `entitlement.changed` marche dans les 2 sens).
4. Aucune attente fixe arbitraire : l'attente porte sur une **condition**.
5. Il est câblé dans un script npm dédié (miroir de `test:chain`) et documenté en une ligne dans le README.
6. Il ne dépend pas de données résiduelles d'un run précédent (idempotent).

## Note

Cette story n'est **pas** un doublon d'AP-07 (« dashboard + e2e chaîne **KYC** ») : AP-07 rejoue la chaîne
**KYC** (dépôt → OCR → revue → approbation). FE-017 rejoue la chaîne **entitlement** (octroi → diffusion →
allumage). Deux boucles distinctes, deux preuves distinctes.
