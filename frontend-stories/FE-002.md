## User Story

En tant que **app cabinet**,
je veux **un client API unique vers la gateway et une couche BFF qui détient le refresh token en cookie httpOnly avec renouvellement silencieux**,
afin que **tous les écrans authentifiés appellent l'API de façon sûre (token jamais exposé au JS), sans reconnexion intempestive, avec une gestion d'erreurs cohérente**.

---

## Contexte

C'est le **socle technique qui débloque toutes les stories authentifiées** (FE-005+). Trois décisions verrouillées le cadrent :

1. **Gateway unique** (Traefik) : une seule URL de base ; routage par préfixe (`/auth`, `/ec`, `/kyc`). En dev, la gateway n'est pas requise → `NEXT_PUBLIC_API_BASE_URL` pointe soit la gateway, soit les ports directs.
2. **Contrats générés depuis l'OpenAPI de la gateway** : les types TS du client sont dérivés du Swagger agrégé (pas de types manuels).
3. **Refresh token en cookie httpOnly** détenu par un **BFF Next.js** (Route Handlers `src/app/api/auth/*`) ; **access token en mémoire** uniquement.

**Pourquoi un BFF ?** Un cookie httpOnly est illisible par le JS (immunité XSS). Le BFF (qui tourne côté serveur Next.js) reçoit le login du client, appelle l'IdP, **capte le `refreshToken` de la réponse JSON** et le pose lui-même en cookie httpOnly sur l'origine du frontend. Le navigateur ne voit jamais le refresh token ; **aucune modification du contrat backend n'est requise**.

L'IdP émet des JWT **RS256** (access 15 min, refresh rotatif 7 j) ; réutilisation d'un refresh consommé → session invalidée (STORY-024). Le client doit être robuste à la rotation.

## Périmètre

**Inclus :**
- **Wrapper `fetch` typé** (`src/lib/api-client.ts`) : base = `NEXT_PUBLIC_API_BASE_URL`, injection `Authorization: Bearer <access>` (depuis la mémoire), sérialisation JSON, parsing d'erreurs normalisé (`{ statusCode, error, message }`), `credentials: 'include'` pour les appels BFF.
- **Génération des types** depuis l'OpenAPI de la gateway (`openapi-typescript`) → `src/types/api.d.ts` ; script `npm run gen:api`.
- **BFF Route Handlers** `src/app/api/auth/` :
  - `POST /api/auth/login` → appelle `POST {API}/auth/login`, reçoit `{ accessToken, refreshToken }`, pose le **cookie httpOnly** `refresh_token` (Secure, SameSite=Strict, `path=/api/auth`, `maxAge` aligné 7 j), renvoie **uniquement** `{ accessToken }` au client.
  - `POST /api/auth/refresh` → lit le cookie, appelle `POST {API}/auth/refresh`, **remplace** le cookie (rotation), renvoie le nouvel `{ accessToken }` ; si l'IdP renvoie 401 (refresh réutilisé/expiré) → efface le cookie, renvoie 401.
  - `POST /api/auth/logout` → appelle `POST {API}/auth/logout` (révocation), efface le cookie.
- **Refresh silencieux** : mécanisme client qui renouvelle l'access token **avant** son expiration (planifié sur l'`exp` décodé, ou à réception d'un 401 → 1 retry après refresh).
- **Provider TanStack Query** (`QueryClient` + `QueryClientProvider`) monté dans le layout ; défauts sensibles (retry, staleTime).
- **Mapping d'erreurs** exploitable par les écrans : 401 (→ refresh puis retry, sinon session expirée), 403 avec codes `KYC_NOT_APPROVED` / `SUBSCRIPTION_REQUIRED` (→ signalés pour redirection onboarding, FR-007), 5xx (→ message générique).

**Hors périmètre :**
- Écran/formulaire de login (UI) → **FE-005** (consomme `POST /api/auth/login`).
- Contexte de session React + guards de route → **FE-005**.
- Écran d'inscription → **FE-004**.

**Décisions & vigilance :**
- **Cookie scope** : `path=/api/auth` limite l'envoi du cookie aux seuls Route Handlers d'auth (moindre exposition). `SameSite=Strict` acceptable car login/refresh sont initiés par l'app elle-même.
- **Rotation & concurrence** : sérialiser les refresh concurrents (une seule requête de refresh en vol, les autres attendent son résultat) pour éviter d'invalider la session via la détection de réutilisation (STORY-024).
- **Décodage `exp`** : décoder l'access token pour planifier le refresh, **sans** faire confiance au contenu pour l'autorisation (la validation reste côté services). Ne pas vérifier la signature côté client.
- **Dev sans gateway** : `/auth/*` en direct sur `:3001` ; en prod via `/auth/*` de la gateway (route publique). Le BFF appelle `${API}/auth/...` dans les deux cas.
- **Ne jamais** mettre de logique métier d'autorisation ici : le front porte le token, les services décident.
