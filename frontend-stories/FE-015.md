## User Story

En tant qu'**administrateur de cabinet (TENANT_ADMIN)**,
je veux **consulter et corriger les informations de mon cabinet**,
afin de **garder une identité à jour — celle-là même que Money Vibes voit pour instruire mon dossier**.

---

## Contexte

**Origine : un gap trouvé le 2026-07-17** en lisant les contrôleurs sur `origin/dev`.
`OrganizationsController` (`@Controller('organizations')`) expose `GET /me` et `PATCH /me` depuis le
cutover — **aucune story frontend ne les couvrait**. L'app cliente n'a donc aujourd'hui aucun écran
pour l'entité qu'elle a pourtant créée à l'inscription (FE-004).

**Pourquoi maintenant, et pourquoi dans ce sprint :** cette story est la **moitié client** de la tranche
verticale du sprint 4. Ce que le cabinet renseigne ici est exactement ce que l'admin lit dans la vue
agrégée des organisations (**AP-02**) pour instruire le KYC (**AP-03**). Les deux moitiés dans le même
sprint = la démo montre la donnée traverser.

**Ajout 2026-07-19 — retrofit de l'identité du shell (données démo en dur).** La vérif LIVE Playwright de
FE-INT-3 a prouvé que le **shell affiche une identité codée en dur**, pas celle de l'API :
- `src/components/layout/app-sidebar.tsx` (l.47-55) : bloc cabinet **« Cabinet Diallo & Associés »** +
  initiales **« CD »** écrits en dur → la sidebar affichait ce nom alors que le compte inscrit était
  « Cabinet Live Test ». Commentaire présent : *« donnée de démonstration jusqu'au câblage de la session (FE-005) »*.
- `src/components/layout/user-menu.tsx` (l.18-24) : constante `DEMO_USER` (`Awa Diallo`,
  `awa.diallo@cabinet-diallo.tg`, `Administrateur`, `AD`) rendue dans le header (avatar, nom, e-mail, rôle) —
  jamais la session réelle, alors que `useSession().user` est déjà disponible juste à côté (app-sidebar l.30).

FE-005 a câblé la **mécanique** de session (token, gardes, logout) mais n'a jamais **remplacé l'affichage démo**.
Le gate d'intégration (FE-INT-1/2/3) a retrofité les features data, **pas le shell**. Puisque FE-015 apporte
déjà `GET /organizations/me`, elle est le bon endroit pour brancher l'identité réelle du shell — et **fermer
le TODO « FE-005 »** laissé dans les deux composants.

## Portée

- **Feature `src/features/organization/`** : `api/get-organization.ts`, `api/update-organization.ts`,
  hooks TanStack Query (`useOrganization`, `useUpdateOrganization` + invalidation).
- **Écran `/parametres/cabinet`** (ou équivalent selon la nav de FE-003) : formulaire React Hook Form + Zod.
- **RBAC** : lecture pour tous les membres ; **modification réservée au `TENANT_ADMIN`** (masquage cohérent
  avec le `nav-config.adminOnly` posé par FE-008 — le masquage reste une commodité, le backend est l'autorité).
- **i18n** : namespace `organization` (FR).
- **États** : chargement / erreur / succès (toast `sonner`), conformes aux composants d'état de FE-003.
- **Retrofit identité du shell (ferme le TODO FE-005) :**
  - `app-sidebar.tsx` : remplacer le bloc **cabinet en dur** (« Cabinet Diallo & Associés » + initiales « CD »)
    par le nom réel issu de `useOrganization()` (`GET /organizations/me`) ; initiales **dérivées** du nom.
    Prévoir chargement (skeleton discret) et dégradation (nom absent / query en erreur → repli neutre, jamais le
    placeholder de démo). ⚠️ Cet appel est **client-side direct** vers auth-service :3001 → dépend de **STORY-109
    (CORS)** pour fonctionner en **navigateur** (au contrat il marche déjà).
  - `user-menu.tsx` : supprimer la constante `DEMO_USER` ; afficher l'utilisateur **réel**. Rôle + id viennent
    déjà de `useSession().user` (JWT) ; **nom complet + e-mail** ne sont pas dans le JWT (`AuthUserDto` = userId,
    tenantId, role, emailVerified) → les lire via **`GET /users/me`** (auth-service). Initiales dérivées du nom.
  - Aucune **donnée d'identité en dur** ne doit subsister dans `src/components/layout/` après cette story.

⚠️ **Ne pas inventer le schéma.** Les champs exacts et leur mutabilité doivent être lus dans
`UpdateOrganizationDto` de l'IdP et dans l'OpenAPI généré (FE-INT-0) — **pas déduits** du formulaire
d'inscription FE-004. Certains champs sont plausiblement en lecture seule (identifiants réglementaires,
statut) : le PATCH sur un champ non whitelisté renvoie **400** (whitelist `class-validator` active).

## Critères d'acceptation

1. L'écran affiche les informations du cabinet issues de `GET /api/v1/organizations/me` (aucune donnée en dur).
2. Un `TENANT_ADMIN` modifie un champ mutable → `PATCH` → succès, toast, données rafraîchies (invalidation).
3. Un `TENANT_USER` voit le profil en lecture seule (aucun contrôle de modification proposé).
4. Les erreurs de validation backend (**400**) sont affichées au niveau du champ concerné.
5. Un champ non whitelisté ou en lecture seule n'est **pas** proposé à l'édition.
6. `401` → refresh silencieux + rejeu ; échec du refresh → redirection `/login?from` (comportement FE-005).
7. Les types viennent de l'**OpenAPI généré** (FE-INT-0), pas d'un type feature-local écrit à la main.
8. Tests : unitaires (schéma Zod, rendu selon rôle) + un test d'intégration du hook (succès + 400).
9. **Shell — cabinet** : la sidebar affiche la **raison sociale réelle** (`GET /organizations/me`), plus jamais
   « Cabinet Diallo & Associés » ni initiales « CD » en dur ; chargement et cas dégradé gérés (repli neutre).
10. **Shell — utilisateur** : le menu utilisateur affiche **nom + e-mail + rôle réels** (session + `GET /users/me`) ;
    la constante `DEMO_USER` est supprimée.
11. **Garde anti-régression** : aucune chaîne d'identité en dur ne subsiste dans `src/components/layout/`
    (grep « Cabinet Diallo », « DEMO_USER », « awa.diallo@ » = 0 occurrence) ; le TODO « FE-005 » des deux
    composants est retiré.
