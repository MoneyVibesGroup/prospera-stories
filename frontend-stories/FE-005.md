## User Story

En tant qu'**utilisateur (gérant ou collaborateur)**,
je veux **me connecter et rester authentifié de façon sûre**,
afin d'**accéder à mon espace cabinet sans me reconnecter en permanence**.

---

## Contexte

Cette story branche l'**UI de session** sur le socle BFF de FE-002 : formulaire de login, contexte de session React, garde des routes `(app)/*`, et déconnexion. Le login passe par le **BFF** (`POST /api/auth/login`) qui pose le cookie httpOnly et renvoie l'access token (gardé en mémoire). Le refresh silencieux est déjà fourni par FE-002 ; ici on l'**exploite** pour hydrater/maintenir la session.

Règles serveur à refléter (FR-003) : identifiants invalides → **401 générique** (pas de distinction e-mail inconnu / mot de passe erroné) ; un utilisateur ou tenant **`SUSPENDED`** ne peut pas se connecter.

---

## Périmètre

**Inclus :**
- Écran `/(auth)/login` : formulaire (e-mail, mot de passe) ; validation Zod ; état de chargement.
- Appel `POST /api/auth/login` (BFF) ; en cas de succès, access token en mémoire + session hydratée.
- **Contexte de session** (`SessionProvider`) : expose l'utilisateur courant (claims non sensibles décodés : `sub`, `org`, `role`, `emailVerified`), l'état `authenticated/loading/unauthenticated`, et `logout()`.
- **Hydratation au démarrage** : au montage de l'app, tenter un `POST /api/auth/refresh` (BFF lit le cookie) → si succès, session restaurée sans login ; sinon état non authentifié.
- **Garde de routes** : les routes sous `(app)/*` redirigent vers `/login` si non authentifié (via layout `(app)` + vérification session, et/ou `middleware.ts` sur présence du cookie).
- **Déconnexion** : `POST /api/auth/logout` (BFF) → cookie effacé + reset session + redirection login.
- Gestion **401 login** générique ; compte **suspendu** → message dédié (le backend renvoie 401 générique ; afficher un message neutre invitant à contacter l'admin si fourni).

**Hors périmètre :**
- Vérification e-mail (landing/renvoi/garde) → **FE-006**.
- Rôles fins par écran (masquage `TENANT_ADMIN`) → traité dans les écrans concernés (FE-008).
- « Mot de passe oublié » : non prévu en phase 1 (hors périmètre PRD).

**Décisions & vigilance :**
- **Deux niveaux de garde** : `middleware.ts` sur la présence du cookie (barrière rapide), **plus** la vérification de session côté layout (le cookie peut exister mais le refresh échouer). Ne pas se fier au seul cookie.
- **Claims décodés = affichage uniquement** : jamais d'autorisation métier côté client ; la source de vérité (suspension, rôles à jour) est backend (read-models). Un écran peut être affiché puis se voir refusé par un 403 — le gérer proprement.
- **SUSPENDED** : le backend renvoie 401 générique ; ne pas inventer un message trop précis. Message neutre + éventuel contact admin.
- **Concurrence de refresh** : déjà sérialisée en FE-002 ; le SessionProvider s'appuie dessus.
- Brancher le bouton « Se déconnecter » du `UserMenu` (FE-003) sur `logout()`.
