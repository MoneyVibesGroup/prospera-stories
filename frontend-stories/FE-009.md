## User Story

En tant que **super-admin de compte**,
je veux **téléverser les justificatifs légaux de mon cabinet (RCCM et carte CFE)**,
afin de **faire valider mon cabinet et débloquer l'accès aux modules métier**.

---

## Contexte

Étape KYC du parcours P1 (FR-005). Le cabinet téléverse **deux documents obligatoires** : **RCCM** et **carte CFE** (PDF/JPEG/PNG, ≤ 10 Mo chacun). Quand les deux sont présents, le tenant passe automatiquement en `UNDER_REVIEW` (backend). Après un rejet, re-soumettre un document crée une **nouvelle version** (l'ancienne devient `SUPERSEDED`) et repasse en `UNDER_REVIEW`.

Le KYC est servi par **`kyc-service`** (`/kyc/*` via gateway). Les fichiers vont dans un **bucket privé** (aucune URL publique). La vérification stricte du type MIME (magic bytes) est **serveur** ; l'UI fait une pré-validation (taille + extension) pour un retour immédiat.

---

## Périmètre

**Inclus :**
- Écran `/(app)/kyc` : deux zones de dépôt (RCCM, CFE) avec sélection de fichier (drag & drop optionnel).
- **Pré-validation client** : taille ≤ 10 Mo, extension ∈ {pdf, jpg/jpeg, png} → message immédiat si non conforme (la vérif de contenu reste serveur).
- **Upload multipart** `POST /kyc/documents` (`type ∈ {RCCM, CFE}` + fichier) avec **indicateur de progression** ; désactivation pendant l'envoi.
- **État par document** : absent / téléversé / en revue / rejeté (avec motif) ; **re-soumission** après rejet (nouvelle version).
- Feedback : succès, erreurs 422 (taille/format côté serveur) mappées, erreur réseau.
- Rappel de l'état global (les deux présents → passage `UNDER_REVIEW`) — détail du stepper en FE-010.

**Hors périmètre :**
- Bandeau d'onboarding global + stepper d'état tenant → **FE-010** (mais cette story affiche l'état par document).
- Revue admin (approve/reject) → hors app (persona plateforme).
- Prévisualisation du document via URL présignée (côté cabinet, non requis par FR-005 ; l'admin la consomme).

**Décisions & vigilance :**
- **Progression d'upload** : `fetch` ne fournit pas nativement la progression d'envoi ; utiliser `XMLHttpRequest` (ou un helper) pour l'upload multipart afin d'exposer `onprogress`, tout en gardant le token/headers du client API.
- **Multipart + BFF** : l'upload va **directement** à la gateway `/kyc/documents` avec le Bearer (pas via le BFF auth) ; s'assurer que le token en mémoire est injecté.
- **Pré-validation ≠ garantie** : le serveur vérifie les magic bytes ; ne pas afficher « validé » sur la seule base de l'extension — juste « téléversé, en attente ».
- **Motif de rejet** : provient de la revue admin (kyc-service) ; l'afficher tel quel, lisible.
- **Types** : le contrat `/kyc/*` vient de l'OpenAPI gateway (FE-002) ; aligner `type` sur l'enum backend (`RCCM`, `CFE`).
