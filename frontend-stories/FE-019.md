## User Story

En tant que **collaborateur ou administrateur d'un cabinet** (`TENANT_USER` / `TENANT_ADMIN`),
je veux **corriger moi-même mon nom, mon téléphone, mon mot de passe et mon adresse e-mail**,
afin de **ne plus dépendre d'un administrateur — ni rester avec le prénom mal orthographié par celui qui m'a invité**.

## Portée

- **Feature `src/features/profile/`** : `api/get-my-profile.ts` (existe peut-être déjà via FE-015 pour le shell —
  **réutiliser, ne pas dupliquer**), `api/update-my-profile.ts`, `api/change-my-password.ts`,
  `api/change-my-email.ts` + hooks TanStack Query (`useMyProfile`, `useUpdateMyProfile`, …) avec
  **invalidation** de la query du profil après succès — la sidebar et le menu utilisateur (FE-015) doivent
  refléter le changement **sans rechargement**.
- **Écran `/parametres/profil`** (ou l'entrée « Mon profil » déjà présente dans le menu utilisateur) —
  **trois blocs distincts**, jamais un seul formulaire :
  1. **Identité** : prénom, nom, téléphone → `PATCH /users/me`.
  2. **Mot de passe** : actuel + nouveau + confirmation → `POST /users/me/password`.
     ⚠️ **Un changement réussi révoque le refresh token** (AC-09 de STORY-123) : prévoir explicitement le
     comportement — message clair puis redirection `/login`, jamais un 401 subi qui ressemble à un bug.
  3. **Adresse e-mail** (STORY-124) : nouvelle adresse + mot de passe actuel → `POST /users/me/email` →
     état **« en attente de confirmation : … »** (lu dans `pendingEmail`) + action **annuler**.
- **RBAC** : **aucun**. Tout membre authentifié édite **son** profil — pas de masquage `adminOnly`, pas de
  variante par rôle. (C'est la différence de fond avec FE-015, réservé au `TENANT_ADMIN`.)
- **i18n** : namespace `profile` (FR).
- **États** : chargement / erreur / succès (toast `sonner`), conformes aux composants d'état de FE-003.

⚠️ **Ne pas inventer le schéma.** Champs et mutabilité se lisent dans `UpdateMyProfileDto`, `ChangePasswordDto`,
`ChangeEmailDto` et `MeResponseDto` de l'**OpenAPI généré** (FE-INT-0) — **pas** déduits de ce document.
En particulier : la **politique de robustesse du mot de passe** doit être la **même** que celle du backend
(STORY-123 impose de la partager avec `RegisterDto`) — la relire côté API, ne pas en réécrire une seconde
qui divergerait.

**Hors périmètre :** profil du **cabinet** (FE-015) · gestion des **autres** membres (FE-008/FE-016) ·
« mot de passe oublié » (surface **non authentifiée**, non planifiée côté backend) · avatar/photo.

## Critères d'acceptation

1. L'écran affiche le profil issu de `GET /api/v1/users/me` — aucune donnée en dur (garde posée par FE-015).
2. Modification de l'identité (prénom / nom / téléphone) → `PATCH` → succès, toast, query invalidée, **et le
   menu utilisateur + la sidebar affichent la nouvelle valeur immédiatement**.
3. Un `TENANT_USER` **non administrateur** accède à l'écran et peut tout y modifier (**aucun** contrôle masqué) —
   test de rendu par rôle qui prouve l'absence de gating.
4. Changement de mot de passe réussi → **204** → message explicite « reconnectez-vous » → redirection `/login`.
   Mot de passe actuel faux → **401** affiché **au niveau du champ concerné**, pas en erreur globale.
5. Erreurs de validation backend (**400**) affichées au niveau du champ ; **429** (throttle) rendu comme tel,
   pas comme une erreur générique.
6. **E-mail** : demande envoyée → l'UI dit explicitement que **l'ancienne adresse reste l'identifiant de
   connexion** jusqu'à confirmation, et affiche `pendingEmail` + le bouton **annuler**.
   ⚠️ Le backend répond **202 même si l'adresse est déjà prise** (anti-énumération, AC-04 de STORY-124) :
   l'UI **ne doit pas** prétendre que l'e-mail est parti à coup sûr — formuler « si cette adresse est
   disponible, un lien de confirmation y a été envoyé ».
7. `401` → refresh silencieux + rejeu ; échec du refresh → redirection `/login?from` (comportement FE-005).
8. Les types viennent de l'**OpenAPI généré** (FE-INT-0), pas d'un type feature-local écrit à la main.
9. Tests : unitaires (schémas Zod des 3 formulaires, rendu identique quel que soit le rôle) + un test
   d'intégration par hook (succès + erreur), dont la **révocation de session** après changement de mot de passe.
10. Le commentaire « écran en lecture seule / demande à formuler » est **retiré** du prototype et de FE-015.
