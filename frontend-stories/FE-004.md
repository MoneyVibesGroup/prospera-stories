## User Story

En tant que **gérant de cabinet**,
je veux **créer le compte de mon cabinet et en devenir l'administrateur**,
afin de **démarrer sur la plateforme PROSPERA**.

---

## Contexte

Premier écran du parcours d'activation P1. Le formulaire crée, côté IdP, une **Organization + un User `TENANT_ADMIN`** de façon atomique (STORY-023). À l'issue, un e-mail de vérification est envoyé (traité en FE-006). L'inscription est une route **publique** de la gateway (`/auth/register`).

Point sensible **sécurité/UX** (FR-001) : un e-mail déjà utilisé renvoie une **erreur générique** (pas de divulgation de l'existence du compte) — le message côté UI ne doit pas distinguer « e-mail déjà pris » d'une autre erreur de validation métier.

---

## Périmètre

**Inclus :**
- Écran `/(auth)/register` : formulaire (raison sociale, pays, prénom, nom, e-mail, mot de passe, confirmation).
- Validation **Zod** miroir des règles serveur : e-mail valide, mot de passe ≥ 8 caractères, confirmation identique, champs requis, pays dans une liste (UEMOA — a minima Togo, Bénin).
- Soumission `POST /auth/register` via le client API ; état de chargement ; désactivation du bouton pendant l'envoi.
- **Succès** : écran/confirmation « Compte créé — vérifiez votre e-mail » (pas de connexion automatique ; la vérification e-mail précède l'accès).
- **Erreurs** : doublon e-mail → message générique non divulgant ; erreurs de validation serveur (400) mappées aux champs ; erreur réseau → toast.
- i18n FR de tous les libellés, messages d'aide et d'erreur.

**Hors périmètre :**
- Envoi/rendu de l'e-mail de vérification et landing → **FE-006**.
- Login → **FE-005**.
- Sélecteur de pays riche (drapeaux, recherche) : un simple select suffit en phase 1.

**Décisions & vigilance :**
- **Message doublon générique** : ne jamais afficher « cet e-mail existe déjà ». Un message neutre type « Impossible de créer le compte avec ces informations » couvre le cas sans divulguer (aligné FR-001 + NFR-001).
- **Register = route publique** : aucun token requis ; l'appel passe par le client API standard, pas par le BFF auth.
- **Confirmation mot de passe** : validée côté client uniquement (le backend ne connaît que le mot de passe).
- Réutiliser `Form` de shadcn (intégration RHF) + `notifyError` de FE-003.
