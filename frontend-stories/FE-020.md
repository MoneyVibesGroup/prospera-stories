## User Story

En tant qu'**utilisateur d'un cabinet** (`TENANT_ADMIN` **ou** `TENANT_USER`),
je veux **retrouver l'accès à mon compte quand j'ai oublié mon mot de passe**,
afin de **ne pas être définitivement exclu de l'outil — et sans avoir à déranger un administrateur**.

---

## Contexte

**Demande explicite du PO (2026-07-22)** : « un utilisateur qui oublie son mot de passe doit être débloqué,
qu'il soit admin ou user pour le cabinet ». Le trou avait été signalé par **STORY-123** puis **vérifié dans le
code** : l'IdP n'a **ni** `forgot-password`, **ni** `reset-password`, **ni** réinitialisation assistée par un
administrateur. Aujourd'hui, un mot de passe oublié = **un compte perdu**.

Le backend est **STORY-125** (sprint 16). FE-020 en est la moitié cliente ; **AP-09** en est la moitié console.
⛔ **Ne pas démarrer avant que STORY-125 soit livrée** — il n'y a rien à appeler.

---

## Portée

- **Écran `/mot-de-passe-oublie`** : un seul champ (adresse e-mail) → `POST /auth/forgot-password`.
- **Écran `/reinitialiser-mot-de-passe?token=…`** : nouveau mot de passe + confirmation →
  `POST /auth/reset-password { token, newPassword }` → succès → redirection `/login` avec un message clair.
- **Entrée depuis `/login`** : lien « Mot de passe oublié ? » (FE-005 ne l'a pas — il n'y avait rien derrière).
- **Feature `src/features/auth/`** : `api/forgot-password.ts`, `api/reset-password.ts` + mutations TanStack
  Query (ce sont des **mutations**, pas des queries — 202/204 sans corps exploitable).
- **i18n** : namespace `auth` (FR) — étendu, pas dupliqué.
- **États** : chargement / erreur / succès conformes aux composants d'état de FE-003.

⚠️ **Ne pas inventer le schéma.** `ForgotPasswordDto` et `ResetPasswordDto` se lisent dans l'**OpenAPI généré**
(FE-INT-0). En particulier, la **règle de robustesse** du nouveau mot de passe est celle du backend (validateur
partagé avec `RegisterDto`, cf. STORY-123/125) : **la relire**, ne pas en réécrire une seconde côté Zod qui
divergerait — sinon l'utilisateur passe le contrôle client et se prend un **400** serveur.

**Hors périmètre :** changement de mot de passe **authentifié** (FE-019) · changement d'e-mail (FE-019/STORY-124)
· le même parcours **côté console admin** (**AP-09**) · toute réinitialisation par un administrateur (écartée
du périmètre backend).

## Critères d'acceptation

1. Depuis `/login`, un lien « Mot de passe oublié ? » mène à `/mot-de-passe-oublie`.
2. Demande envoyée → l'écran affiche un message **volontairement non affirmatif** :
   ⚠️ **« Si un compte existe pour cette adresse, un lien vient d'y être envoyé. »**
   Le backend répond **202 même pour une adresse inconnue** (anti-énumération, AC-05 de STORY-125) :
   l'UI **ne doit ni confirmer ni infirmer** l'existence du compte. **Aucun** message du type « e-mail envoyé à
   … » ou « aucun compte trouvé ».
3. Le formulaire de demande ne révèle rien non plus par son **comportement** : même écran de succès, même délai
   perçu, pas de champ qui vire au rouge selon l'adresse.
4. `/reinitialiser-mot-de-passe?token=…` : token **absent** de l'URL → écran d'erreur explicite, aucun appel réseau.
5. Réinitialisation réussie → **204** → message « Mot de passe modifié, connectez-vous » → redirection `/login`.
6. Token **invalide, expiré ou déjà utilisé** → **400** → message clair proposant de **refaire une demande**
   (lien vers `/mot-de-passe-oublie`) — jamais une erreur générique sans issue.
7. Mot de passe trop faible → **400** affiché **au niveau du champ**, avec la règle **du backend**.
8. **429** (throttle) rendu comme tel (« trop de tentatives, réessayez dans quelques minutes »), pas comme une
   panne.
9. Les deux écrans sont **publics** : accessibles **sans session**, et **non redirigés** par la garde de routes
   de FE-005 (piège classique — un middleware qui pousse vers `/login` rendrait la page inatteignable
   précisément quand on en a besoin). **Test dédié.**
10. Les types viennent de l'**OpenAPI généré** (FE-INT-0), pas d'un type feature-local écrit à la main.
11. Tests : unitaires (schémas Zod des 2 formulaires, rendu des cas 400/429, token manquant) + un test
    d'intégration par mutation (succès + erreur) + **le test d'accessibilité sans session** (AC-9).

## Notes

- **Le lien du mail pointe vers CETTE application** (`PASSWORD_RESET_URL_TENANT`, config **serveur** de
  STORY-125) — le front **ne déclare jamais** sa propre URL de retour : ce serait une redirection ouverte, et
  le backend rejette tout champ de ce type (**400**, AC-04 de STORY-125). Ne pas essayer de « passer un
  `redirectUrl` », c'est refusé **par conception**.
- L'app cliente se sert sur **:3100** (origine autorisée par le CORS de STORY-109) — dev/start et Playwright.
- **Piège TanStack connu** (FE-INT-4) : une `queryFn` qui résout `undefined` met la query **en erreur**. Ici
  tout passe par des **mutations**, mais rester vigilant sur le typage des réponses **sans corps** (204).
