## User Story

En tant qu'**utilisateur inscrit**,
je veux **valider mon adresse e-mail via un lien et pouvoir en redemander un**,
afin de **prouver que je contrôle cette adresse et débloquer l'accès à la plateforme**.

---

## Contexte

La vérification e-mail est le verrou entre l'inscription (FE-004) et l'accès aux écrans authentifiés (FR-002) : tout endpoint authentifié (hors verify/resend/logout) est refusé tant que l'e-mail n'est pas vérifié. L'utilisateur arrive sur la landing via le **lien reçu par e-mail** (`/verify-email?token=...`) ; le token est à usage unique, TTL 24 h. Le renvoi est limité côté serveur (3/h).

Deux surfaces : (1) la **landing** publique qui consomme le token, (2) la **garde + bannière** dans l'espace authentifié qui invite à vérifier et propose le renvoi.

---

## Périmètre

**Inclus :**
- **Landing** `/(auth)/verify-email` : lit `token` du query ; appelle `GET /auth/verify-email?token=` ; affiche succès / expiré / déjà consommé / invalide ; CTA vers login (succès) ou renvoi (échec, si session).
- **Renvoi** : `POST /auth/resend-verification` (utilisateur authentifié non vérifié) ; feedback ; gestion de la limite serveur (3/h) → message clair si dépassée.
- **Bannière + garde** dans `(app)` : si `emailVerified = false` (claim de session), afficher une bannière persistante « Vérifiez votre e-mail » avec bouton renvoyer ; restreindre l'accès aux écrans authentifiés (rediriger vers un écran « vérification requise » ou limiter la navigation), en laissant accessibles logout et le renvoi.

**Hors périmètre :**
- Envoi effectif de l'e-mail (backend / Mailhog).
- Acceptation d'invitation (l'invité est réputé vérifié à l'acceptation) → **FE-007**.

**Décisions & vigilance :**
- **`emailVerified` vient des claims de session** (décodés en FE-005) pour l'affichage/garde ; la source de vérité reste le backend — après vérification, forcer un refresh de session pour rafraîchir le claim.
- **Rafraîchir le claim** : la vérification se fait souvent hors session (lien e-mail cliqué dans un autre contexte). Prévoir que l'utilisateur devra peut-être se reconnecter/refresh pour que `emailVerified` bascule.
- **Limite 3/h** : le backend renvoie une erreur de rate limit (429) → message dédié « réessayez plus tard », pas un toast d'erreur générique.
- Landing = route **publique** (le token suffit) ; le renvoi nécessite une **session**.
