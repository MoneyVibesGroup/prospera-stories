## User Story

En tant qu'**utilisateur du cabinet**,
je veux **une interface cohérente (en-tête, navigation, zones de contenu, retours visuels)**,
afin de **me repérer dans l'application et comprendre l'état de mes actions (chargement, succès, erreur)**.

---

## Contexte

Cette story livre l'**enveloppe visuelle** partagée par tous les écrans : deux layouts (public pour l'auth, applicatif pour l'espace cabinet authentifié), la navigation, et les **primitives de feedback** (toasts, états de chargement/erreur) réutilisées partout. Elle ne branche aucune API — elle prépare les emplacements que FE-004+ rempliront.

Les groupes de routes App Router `(auth)` et `(app)` matérialisent la séparation public/authentifié (la garde d'accès réelle vient avec FE-005).

---

## Périmètre

**Inclus :**
- **Layout public** `(auth)` : structure épurée (logo, carte centrée) pour inscription/login/vérif/invitation.
- **Layout applicatif** `(app)` : app shell (en-tête avec identité cabinet + menu utilisateur/déconnexion, navigation latérale ou supérieure vers Utilisateurs / KYC / Abonnement, zone de contenu).
- **Navigation** : items de menu (i18n FR) avec état actif ; responsive (mobile : menu repliable).
- **Primitives de feedback** : composant Toaster (shadcn `sonner` ou `toast`) monté globalement ; helpers `notifySuccess/notifyError`.
- **États réutilisables** : composants `LoadingState`, `ErrorState` (avec retry), `EmptyState` — utilisés par les listes/écrans à venir.
- **Thème** : tokens shadcn (couleurs, radius) ; support clair/sombre optionnel via classe (a minima ne pas casser le thème système).

**Hors périmètre :**
- Garde d'accès réelle (redirection si non authentifié) → **FE-005**.
- Bandeau d'onboarding / état tenant → **FE-010**.
- Tout contenu d'écran métier.
