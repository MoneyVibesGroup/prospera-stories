## User Story

En tant que **super-admin de compte au KYC approuvé**,
je veux **payer mon abonnement via FedaPay (Mobile Money ou carte) et voir le résultat**,
afin d'**activer l'accès aux modules métier de mon cabinet**.

---

## Contexte

Cœur de la monétisation (FR-009). `POST /billing/checkout` crée une transaction FedaPay et renvoie une **URL de paiement** ; l'app **redirige** l'utilisateur vers FedaPay. Une souscription `PENDING_PAYMENT` est créée localement. Après paiement, l'utilisateur **revient** sur l'app (page de retour) — mais l'**activation réelle est asynchrone** côté backend : elle dépend du **webhook signé + idempotent** (FR-010, STORY-017) et d'une re-vérification via l'API FedaPay. **Le retour utilisateur ne fait donc pas foi** ; l'app doit **sonder** l'état d'abonnement pour confirmer l'activation.

Garde (FR-007) : KYC non approuvé → 403 `KYC_NOT_APPROVED` sur le checkout. Un abonnement déjà actif pour la même période → 409 (empêcher le double checkout).

---

## Périmètre

**Inclus :**
- **Lancement du checkout** : depuis le plan choisi (FE-011), `POST /billing/checkout` → réception de l'URL FedaPay → **redirection** vers FedaPay.
- **Souscription `PENDING_PAYMENT`** reflétée dans l'UI (état transitoire).
- **Page de retour** `/(app)/billing/return` (succès/échec/abandon selon le retour FedaPay) : message d'attente « paiement en cours de confirmation ».
- **Sondage d'activation** : après le retour, poller `GET /billing/subscription` (backoff borné) jusqu'à `ACTIVE` (webhook traité) ou timeout → message « confirmation en cours, revenez plus tard » (l'activation finira via webhook).
- **Gestion d'erreurs** : 403 `KYC_NOT_APPROVED` → orientation onboarding (FE-010) ; 409 abonnement déjà actif → message + lien vers l'abonnement (FE-013) ; échec/abandon → possibilité de **relancer** un checkout.
- Invalidation des queries `subscription` et `tenant` à l'activation confirmée.

**Hors périmètre :**
- Traitement du webhook / activation (backend, FR-010).
- Historique des transactions & cycle de vie → **FE-013**.
- Renouvellement automatique / prélèvement récurrent (hors périmètre PRD phase 1).

**Décisions & vigilance :**
- **Ne jamais activer côté client** : l'activation est backend (webhook + re-vérif API FedaPay, NFR-003). L'UI **observe** l'état, elle ne le décide pas.
- **Redirection** : rediriger vers l'URL FedaPay renvoyée (navigation top-level). Prévoir l'`callback_url`/retour configuré côté backend pointant vers `/billing/return`.
- **Sondage borné** : limiter le nombre de tentatives et la durée (ex. ~30-60 s) ; au-delà, message rassurant — l'abonnement s'activera via webhook, visible ensuite dans FE-013/FE-010.
- **Idempotence UX** : gérer proprement le 409 (déjà actif) — ne pas laisser l'utilisateur relancer un paiement inutile.
- **Sandbox/live** : transparent pour le front (`FEDAPAY_ENVIRONMENT` est backend) ; aucune logique de clé côté client.
